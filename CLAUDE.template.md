## ⚙ 자동 분석 지시문

> **이 파일에 `{{` 플레이스홀더가 남아 있으면, 대화 시작 시 아래 절차를 자동 실행할 것.**

### 분석 절차
1. **프로젝트 스캔** — 루트 디렉터리의 파일/폴더 구조를 탐색
2. **기술 스택 감지** — `package.json`, `composer.json`, `requirements.txt`, `Gemfile`, `go.mod`, `pom.xml`, `build.gradle`, `Cargo.toml`, `docker-compose.yml`, `.env` 등의 설정 파일로 언어·프레임워크·인프라 파악
3. **코드 패턴 분석** — 소스 파일을 샘플링하여 파일 명명 규칙, include/import 패턴, DB 접근 패턴, 응답 패턴, 인증 방식 등을 파악
4. **보안 패턴 수집** — SQL 쿼리 방식, 출력 이스케이프, CSRF 처리, 파일 업로드 로직 등을 소스에서 추출
5. **플레이스홀더 채우기** — 분석 결과로 `{{}}` 부분을 실제 값으로 교체
6. **사용자 확인** — 채운 내용을 사용자에게 보여주고, 승인 후 CLAUDE.md에 저장

### 감지 우선순위
| 파일 | 감지 대상 |
|------|----------|
| `package.json` | Node.js, 프레임워크(React/Vue/Next 등), 스크립트 명령어 |
| `composer.json` | PHP, 프레임워크(Laravel 등), ORM |
| `requirements.txt` / `pyproject.toml` | Python, 프레임워크(Django/Flask/FastAPI 등) |
| `Gemfile` | Ruby, Rails 등 |
| `go.mod` | Go, 모듈 의존성 |
| `pom.xml` / `build.gradle` | Java/Kotlin, Spring 등 |
| `Cargo.toml` | Rust |
| `docker-compose.yml` | 컨테이너 구성, 포트, DB 종류 |
| `Makefile` / `Dockerfile` | 빌드/실행 명령어 |
| `.env.example` / `.env` | 환경변수 키 목록 (값 제외) |
| DB 설정 파일 / ORM 설정 | DB 종류, 테이블 명명 규칙 |
| 소스 코드 샘플 (최대 10개) | 파일 명명, include 패턴, 응답 패턴, 보안 패턴 |

### 주의사항
- `.env` 파일의 **값(시크릿)**은 절대 CLAUDE.md에 기록하지 않을 것. 키 이름만 참조
- 분석 결과가 불확실한 항목은 `{{}}` 를 유지하고 사용자에게 질문할 것
- 이미 채워진 섹션(플레이스홀더 없음)은 건드리지 않을 것

## 6. QA 핵심 규칙

- QA 파일명: `qa_YYYYMMDD_v#.md` (예: `qa_20260318_v1.md`)
- 주제별: `qa_YYYYMMDD_v#_주제.md`
- 작업 시작 시 `QA/index.md` 생성, 보고서 추가 시 갱신 필수
- 상세 형식: `QA/CLAUDE.md` 참조

## 7. SKILL 핵심 규칙

- 명세서 파일명: `NN_섹션명.md` (예: `01_DASHBOARD.md`)
- 변경 요청: `CHANGE_MMDD.md` (예: `CHANGE_0317.md`)
- 첫 작업 시 `00_OVERVIEW.md` 먼저 생성, `index.md` 갱신 필수
- 상세 형식: `SKILL/CLAUDE.md` 참조
