# claude-config — 하네스 엔지니어링 CLAUDE.md 관리

Claude Code에서 사용하는 CLAUDE.md 파일을 하네스 엔지니어링 원칙에 따라 계층별로 관리하는 레포지터리입니다.

> ⚠️ **이 레포의 `CLAUDE.md`는 복사하지 마세요!**
> 루트의 `CLAUDE.md`는 이 레포(claude-config) 자체를 작업할 때 사용되는 설정입니다.
> 새 프로젝트에는 **`CLAUDE.template.md`** 를 복사하세요.

---

## 빠른 시작

### 1. 글로벌 설정 적용 (모든 프로젝트 공통)

```bash
# 공통 규칙만 적용
cp GLOBAL/CLAUDE.md ~/.claude/CLAUDE.md

# 언어별 보안 규칙 병합 (PHP 프로젝트 예시)
cat GLOBAL/CLAUDE.md GLOBAL/security-php.md > ~/.claude/CLAUDE.md
```

### 2. 새 프로젝트에 적용

```bash
# ⚠️ CLAUDE.md가 아닌 CLAUDE.template.md를 복사!
cp CLAUDE.template.md /path/to/project/CLAUDE.md

# QA/SKILL 디렉터리는 빈 디렉터리로 생성 (필요 시)
mkdir -p /path/to/project/QA /path/to/project/SKILL
```

Claude Code를 시작하면 `CLAUDE.template.md`의 플레이스홀더(`{{}}`)를 감지하여 소스 코드를 자동 분석합니다.
플레이스홀더를 채운 후에는 자동 분석 지시문 섹션을 삭제하세요.

### 3. CI 검증 스크립트 복사 (선택)

```bash
# 프로젝트에 CI 검증을 적용하려면
cp -r scripts/ /path/to/project/scripts/
cp -r .github/ /path/to/project/.github/
cp -r .claude/ /path/to/project/.claude/
```

## 파일 설명 — 무엇을 복사하고 무엇을 복사하지 말 것인가

| 파일 | 용도 | 복사 대상? |
|------|------|-----------|
| `CLAUDE.md` | **이 레포 전용** 설정. claude-config 레포를 작업할 때만 사용 | ❌ 복사 금지 |
| `CLAUDE.template.md` | 새 프로젝트 초기화 템플릿. 플레이스홀더(`{{}}`) 포함 | ✅ `CLAUDE.md`로 복사 |
| `GLOBAL/CLAUDE.md` | 글로벌 공통 규칙 원본 | ✅ `~/.claude/CLAUDE.md`로 복사 |
| `GLOBAL/security-*.md` | 언어별 보안 규칙 | ✅ 글로벌에 병합 |
| `.claude/commands/` | 커맨드 (선택적 효율 도구) | ✅ 선택 복사 |
| `scripts/` | CI 검증 스크립트 | ✅ 선택 복사 |
| `.github/workflows/` | PR 자동 검증 CI | ✅ 선택 복사 |

## 하네스 엔지니어링 3계층 구조

```
┌─────────────────────────────────────────────────┐
│  Layer 1: 지시                                   │
│  루트 CLAUDE.md, 글로벌 CLAUDE.md               │
│  → 에이전트가 세션 시작 시 자동 로딩              │
│  → 핵심 규칙만 직접 포함 (참조 링크 아님)         │
├─────────────────────────────────────────────────┤
│  Layer 2: 강제 주입 (선택적 효율 도구)            │
│  .claude/commands/                              │
│  → 유저가 커맨드 사용 시 규칙이 컨텍스트에 주입    │
│  → 없어도 CI가 잡아줌, 있으면 첫 시도부터 맞춤    │
├─────────────────────────────────────────────────┤
│  Layer 3: 검증                                   │
│  CI workflow, scripts/                          │
│  → 에이전트가 규칙을 어기면 머지 차단             │
│  → 유저 의지와 무관하게 강제 적용                 │
└─────────────────────────────────────────────────┘
```

## 토큰 효율 설계 원칙

| 원칙 | 설명 |
|------|------|
| **자동 로딩 위치에만 적는다** | 루트·글로벌 CLAUDE.md만 자동 로딩 보장 |
| **"참조하라" 지시는 쓰지 않는다** | 에이전트가 읽을지 보장 불가 |
| **핵심만 압축해서 직접 포함한다** | 토큰 효율 = 파일 분리가 아니라 압축 |
| **상세 형식 검증은 CI가 강제한다** | 에이전트가 몰라도 머지 차단 |

## CLAUDE.md 계층 구조

실제 프로젝트에 적용했을 때의 계층:

```
~/.claude/CLAUDE.md          ← 글로벌 (모든 프로젝트에 적용, 자동 로딩 ✅)
  └ 원본: GLOBAL/CLAUDE.md    ← 이 레포에서 버전 관리
/CLAUDE.md                   ← 프로젝트 루트 (자동 로딩 ✅, 핵심 규칙 직접 포함)
  └ 원본: CLAUDE.template.md  ← 이 레포의 템플릿
```

> 에이전트가 읽는 것은 루트 CLAUDE.md와 글로벌 CLAUDE.md **두 곳뿐**입니다.
> 하위 디렉터리의 CLAUDE.md는 자동 로딩이 보장되지 않으므로, 에이전트에게 필요한 핵심 규칙은 반드시 루트 CLAUDE.md에 직접 포함해야 합니다.

## 디렉터리 구조

```
claude-config/
├── CLAUDE.md              # 이 레포 전용 규칙 (복사 금지 ❌)
├── CLAUDE.template.md     # 새 프로젝트 초기화용 템플릿 (복사 대상 ✅)
├── GLOBAL/
│   ├── CLAUDE.md          # 글로벌 공통 규칙 (~/.claude/CLAUDE.md 원본)
│   ├── security-php.md    # PHP 언어별 보안 규칙
│   ├── security-js.md     # JavaScript 언어별 보안 규칙
│   ├── security-python.md # Python 언어별 보안 규칙
│   └── security-java.md   # Java 언어별 보안 규칙
├── .claude/
│   ├── settings.json      # 도구 허용/차단 설정
│   └── commands/
│       ├── qa-audit.md    # QA 보안 감사 커맨드
│       ├── skill-write.md # SKILL 명세서 작성 커맨드
│       └── security-check.md # 보안 체크리스트 커맨드
├── scripts/
│   ├── validate-qa-naming.sh        # QA 명명 규칙 검증
│   ├── validate-skill-naming.sh     # SKILL 명명 규칙 검증
│   └── validate-security-patterns.sh # 보안 패턴 위반 탐지
├── .github/
│   └── workflows/
│       └── harness-check.yml        # PR 시 자동 검증 CI
└── README.md              # 이 파일
```

## 커맨드 사용법

커맨드는 **선택적 효율 도구**입니다. 없어도 CI가 규칙 위반을 잡아주지만, 있으면 첫 시도부터 규칙에 맞는 결과를 얻을 수 있습니다.

| 커맨드 | 용도 |
|--------|------|
| `/qa-audit [대상]` | QA 보안 감사 절차 + 보고서 형식 주입 |
| `/skill-write [기능명]` | SKILL 명세서 작성 절차 + 형식 주입 |
| `/security-check [대상]` | 보안 체크리스트 실행 |

## CI 검증

PR 생성 시 `.github/workflows/harness-check.yml`이 자동 실행됩니다:

1. **QA 명명 규칙** — `qa_YYYYMMDD_v#.md` 등 패턴 검증
2. **SKILL 명명 규칙** — `NN_섹션명.md` 등 패턴 검증
3. **보안 패턴 탐지** — eval(), SQL 직접 삽입, .env 추적 등 탐지

위반 시 머지 차단됩니다.

### 로컬 실행

```bash
bash scripts/validate-qa-naming.sh QA
bash scripts/validate-skill-naming.sh SKILL
bash scripts/validate-security-patterns.sh .
```

## QA 디렉터리 상세 가이드

> 이 섹션은 QA 작업의 상세 형식과 절차를 설명합니다.
> 에이전트에게 필요한 핵심 규칙(명명 규칙 등)은 `CLAUDE.template.md`에 이미 포함되어 있습니다.

### 목적

QA 디렉터리는 프로젝트의 품질 보증(QA) 산출물을 관리합니다.
초기에는 비어 있으며, QA 진행에 따라 아래 규칙대로 파일을 생성해 나갑니다.

### QA 진행 워크플로우

```
[1] 보안 감사 → [2] 조치 → [3] 테스트 작성·실행 → [4] 배포 → [5] 검증
```

### 파일 명명 규칙

| 유형 | 파일명 패턴 | 예시 |
|------|------------|------|
| QA 이력 마스터 | `index.md` | 전체 QA 현황 요약, 보고서 링크 모음 |
| 보안감사 보고서 | `qa_YYYYMMDD_v#.md` | `qa_20260318_v1.md` |
| 주제별 보고서 | `qa_YYYYMMDD_v#_주제.md` | `qa_20260318_v2_db_tuning.md` |
| 레거시 체크리스트 | `legacy_checklist.md` | 레거시 프로젝트 초기 점검용 |
| E2E 테스트 스크립트 | `e2e-범위.*` | `e2e-test.mjs`, `e2e-security.py` 등 |
| 비주얼 테스트 스크립트 | `visual-test.*` | 스크린샷 비교 |
| 테스트 리포트 | `범위-report.html` | `e2e-report.html` |
| 배포 스크립트 | `deploy.*` / `deploy_모듈명.*` | `deploy.sh`, `deploy.py` |
| 스크린샷 | `screenshots/` | 비주얼 테스트 베이스라인 |
| 배포 백업 | `backups/YYYYMMDD_HHMMSS/` | 배포 전 자동 백업 |

- 상태 아이콘: ✅ 완료 / 🔄 진행중 / ❌ 실패 / ⬜ 대기

### E2E 테스트 작성 지침

필요 시 프로젝트 환경에 맞는 E2E 테스트를 직접 작성합니다.

**작성 원칙:**
- 테스트 데이터에는 `[E2E_TEST]` 접두사를 붙여 실제 데이터와 구분
- 테스트별 결과를 수집하여 리포트로 출력
- 가능하면 headless / headed 모드 모두 지원

**테스트 분류:**

| 분류 | 목적 | 파일명 예시 |
|------|------|------------|
| 기능 CRUD | 핵심 화면별 생성/수정/삭제 검증 | `e2e-test.*` |
| 전체 커버리지 | 전 메뉴/페이지 테스트 | `e2e-full.*` |
| 보안 | 비로그인 차단, CSRF 검증, XSS 필터 | `e2e-security.*` |
| 비주얼 리그레션 | 스크린샷 비교로 UI 변경 감지 | `visual-test.*` |

### 배포 스크립트 작성 지침

| 모드 | 설명 |
|------|------|
| `full` | 소스 + DB 전체 배포 |
| `src` | 소스만 배포 |
| `db` | DB만 배포 |
| `diff` | VCS diff 기반 변경분만 배포 |
| `rollback` | 백업에서 복원 |

**필수 요구사항:**
- 배포 전 원격 백업 자동 생성 (`backups/YYYYMMDD_HHMMSS/`)
- 배포 후 핵심 페이지/엔드포인트 상태 자동 검증
- 확인 프롬프트 (y/N) 후 실행

### 레거시 프로젝트 체크리스트

레거시 프로젝트 인수 시 `legacy_checklist.md`를 작성하여 5단계 점검:

| 단계 | 내용 |
|------|------|
| 1. 환경 구축 | 런타임/DB 설정, 의존성 설치, .env 분리 |
| 2. 소스 에러 | 문법 오류, 런타임 버전 호환, deprecated API 수정 |
| 3. 보안 감사 | 백도어, 파일업로드, SQLi, XSS, CSRF, 세션, 인증 |
| 4. DB 튜닝 | 인덱스, N+1, 페이지네이션, 슬로우쿼리 |
| 5. 운영 준비 | 모니터링, 배포, 백업 자동화 |

## SKILL 디렉터리 상세 가이드

> 이 섹션은 SKILL 명세서의 상세 형식과 절차를 설명합니다.
> 에이전트에게 필요한 핵심 규칙(명명 규칙 등)은 `CLAUDE.template.md`에 이미 포함되어 있습니다.

### 목적

SKILL 디렉터리는 Figma 디자인 → 분석 문서 → 개발 → 검증까지의 전체 워크플로우 산출물을 관리합니다.
초기에는 비어 있으며, 워크플로우 진행에 따라 아래 규칙대로 파일을 생성해 나갑니다.

### 워크플로우 개요

```
[1] Figma 스크린샷 수집
 → [2] 스크린샷별 분석 MD 생성 + index.md 정리
  → [3] 개발 중 디자인 검증 (스크린샷·MD 재확인)
   → [4] 디자인 가이드/퍼블리싱 참고 반영
    → [5] 퍼블 ↔ 개발소스 일치도 점수화
```

### 파일 명명 규칙

**스킬 문서:**

| 파일명 패턴 | 용도 | 예시 |
|------------|------|------|
| `00_OVERVIEW.md` | 프로젝트 전체 IA, 파일 매핑, 공통 패턴 | 항상 최초 생성 |
| `NN_섹션명.md` | 메뉴/기능별 명세서 (NN = 메뉴번호) | `01_DASHBOARD.md`, `08_CUSTOMER_CENTER.md` |
| `CHANGE_MMDD.md` | 기존 기능 변경 요청 명세 | `CHANGE_0317.md`, `CHANGE_0415.md` |
| `DB_SCHEMA.md` | DB 스키마 정의, ER 다이어그램, 명명 규칙 | 테이블/컬럼 전체 정리 |
| `index.md` | 마일스톤 트래커 | 화면 ↔ 파일 ↔ 상태 매핑 |
| `migration_*.sql` | DB 스키마 변경 SQL (필요 시) | `migration_faq_cate.sql` |

**사용자(프론트) 문서 — 관리자/사용자 구분이 있는 프로젝트에서만 사용:**

| 파일명 패턴 | 용도 | 예시 |
|------------|------|------|
| `user_index.md` | 사용자 화면 트래커 | 퍼블 ↔ 개발 매핑 + 상태 |
| `USER_NN_섹션명.md` | 화면별 퍼블·백엔드 연동 명세 | `USER_01_MAIN.md`, `USER_07_MYPAGE.md` |

### 파일 생성 전 필수 체크리스트

SKILL 디렉터리에 파일을 생성·수정하기 전에 반드시 확인:

1. **문서 유형 판별** — `NN_섹션명.md` / `CHANGE_*.md` / `00_OVERVIEW.md` 등 중 어디에 해당?
2. **필수 섹션 확인** — 해당 유형의 MUST 섹션이 모두 포함되었는가?
3. **형식 우선순위** — 기존 파일보다 이 가이드의 템플릿 형식을 따른다
4. **index.md 갱신** — 파일 생성/수정 후 반드시 갱신

### Step 1. Figma 스크린샷 수집

Figma REST API (또는 MCP get_screenshot)로 기획서·디자인의 모든 프레임을 스크린샷으로 추출합니다.

> **get_metadata로 하위 노드를 탐색할 수 없는 경우가 많으므로, `use_figma`를 우선 사용한다.**

1. `use_figma`로 페이지·프레임 목록 추출
2. 프레임별 상세 조회 (`get_design_context`, `get_screenshot`)
3. 전체 종합 정리

### Step 2. 스크린샷별 분석 MD 생성

각 화면(프레임)을 분석하여 스킬 문서를 작성합니다. 위 "문서 작성 형식"을 참고하세요.

### Step 3. 개발 중 디자인 검증

디자인과 다르게 구현하고 있다고 판단될 때:
1. 해당 화면의 스킬 문서 재확인
2. Figma 스크린샷 재캡처하여 비교
3. 불일치 항목을 구체적으로 명시하고 수정

### Step 4. 디자인 가이드 · 퍼블리싱 참고

- 퍼블리싱 소스가 있으면 USER_NN 문서에 **퍼블 파일 ↔ 개발 파일 매핑** 기록
- 퍼블리싱의 헤더/푸터/메뉴 구조, 사용 라이브러리를 문서에 명시

### Step 5. 퍼블 ↔ 개발소스 일치도 점수화

| 항목 | 배점 | 설명 |
|------|------|------|
| 레이아웃 구조 | 25 | HTML 구조, 배치 일치도 |
| 스타일 정확도 | 25 | 색상, 폰트, 간격, 크기 등 |
| 반응형 대응 | 20 | 브레이크포인트별 레이아웃 |
| 인터랙션 구현 | 15 | 호버, 클릭, 트랜지션 |
| 접근성 · 시맨틱 | 15 | 시맨틱 태그, ARIA |

점수 기준: 90~100 퍼블과 동일 / 70~89 미세 조정 필요 / 50~69 디테일 차이 다수 / 50 미만 재작업
