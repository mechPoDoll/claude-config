# QA 보안 감사 커맨드

$ARGUMENTS 대상에 대해 보안 감사를 수행합니다.

> **사용 예시:**
> - `/qa-audit src/auth/` — 인증 모듈 대상 보안 감사
> - `/qa-audit 결제 기능` — 결제 관련 파일 탐색 후 점검
> - `/qa-audit https://example.com` — 해당 사이트 소스 기준 점검
> - `/qa-audit 전체` — 프로젝트 전수 점검

## 파일 명명 규칙 (반드시 준수)

| 유형 | 위치 | 파일명 패턴 | 예시 |
|------|------|------------|------|
| QA 이력 마스터 | `QA/` | `index.md` | 전체 QA 현황 요약 |
| 보안감사 보고서 | `QA/` | `qa_YYYYMMDD_v#.md` | `qa_20260318_v1.md` |
| 주제별 보고서 | `QA/` | `qa_YYYYMMDD_v#_주제.md` | `qa_20260318_v2_db_tuning.md` |
| 검증 결과 | `QA/` | `qa_YYYYMMDD_v#_*_verify_results.md` | `qa_20260429_v3_refactor_verify_results.md` |
| 레거시 체크리스트 | `QA/` | `legacy_checklist.md` | 레거시 초기 점검용 |
| 검증 SQL (도구) | **`sql/verify/`** | `{영역}_verify.sql` | `refactor_0427_v1_v2_verify.sql` |

### 산출물 위치 규칙 (중요)

**`QA/` 디렉터리는 `.md` 보고서만 둔다.** 검증용 SQL 파일은 `sql/verify/` 하위에 배치한다.

| 산출물 | 위치 | 이유 |
|--------|------|------|
| QA 보고서 (.md) | `QA/` | QA 명명 규칙 검증 통과 (`scripts/validate-qa-naming.sh`) |
| 검증 SQL (.sql) | `sql/verify/` | QA/ 명명 규칙은 .sql 비허용 + DB 도구는 sql/ 트리에 통일 |
| 검증 결과 (.md) | `QA/` | SQL 실행 stdout을 누적 보존, 짝 SQL 파일을 `../sql/verify/`로 링크 |

**SQL/MD 분리 원칙**: SQL = "어떻게 검증하는지" (재현 도구), MD = "그때 통과했는지" (시점 기록). 두 파일을 합치면 재현성·기록성 동시에 잃는다.

**보존 정책**: 검증 SQL은 QA 종료 후에도 회귀 검증 자산으로 영구 보존. 동일 영역 후속 작업 시 재실행하여 데이터 정합성 비교에 사용. 임의 삭제 금지.

## 보안 감사 절차

### Step 0. 대상/범위 확인 (필수)
$ARGUMENTS에서 QA 대상과 범위를 식별한다:

| 대상 유형 | 확인 방법 |
|----------|----------|
| **파일/디렉터리** | 지정된 경로의 소스 코드를 스캔 |
| **기능/모듈** | 해당 기능 관련 파일을 탐색하여 범위 결정 |
| **사이트 URL** | 해당 사이트의 소스를 기준으로 점검 |
| **전체 프로젝트** | "전체" 명시 시 루트부터 전수 점검 |

**대상이 지정되지 않으면** 사용자에게 질문한다:
> "QA 보안 감사를 진행하려면 대상을 지정해 주세요. 아래 중 선택해 주세요:"
> 1. 특정 파일/디렉터리 경로 (예: `src/auth/`, `login.php`)
> 2. 특정 기능 또는 모듈명 (예: `회원가입`, `결제`)
> 3. 사이트 URL (해당 사이트 소스 기준)
> 4. 프로젝트 전체
>
> 대상 없이 진행하면 점검 범위가 불명확하여 누락이 발생할 수 있습니다.

### Step 1~5. 점검 실행
1. `QA/index.md`가 없으면 먼저 생성한다
2. 대상 파일/디렉터리를 스캔하여 아래 보안 항목을 점검한다
3. **Playwright 자동 검증 여부를 사용자에게 질문한다:**
   > "Playwright를 사용하여 UI/기능을 자동 검증할까요? (브라우저 테스트가 가능한 환경인 경우 권장)"
   > - Yes → Playwright로 주요 화면 접근·폼 제출·인증 흐름 등을 자동 테스트하고 결과를 보고서에 포함
   > - No / 환경 미지원 → 코드 정적 분석만으로 점검 진행
4. 오늘 날짜 기준 `qa_YYYYMMDD_v1.md` 파일명으로 보고서를 작성한다
5. `QA/index.md`에 새 보고서를 추가한다

## 점검 항목 (공통)

- [ ] 사용자 입력 → SQL 문자열 직접 삽입 (SQL Injection)
- [ ] 이스케이프 없는 사용자 데이터 출력 (XSS)
- [ ] CSRF 토큰 미적용 상태 변경 요청
- [ ] 인증 체크 없는 처리 엔드포인트
- [ ] 평문 비밀번호 저장
- [ ] `.env` / 시크릿 파일 커밋 여부
- [ ] 운영 환경 상세 에러 노출
- [ ] 사용자 입력 → 셸 명령 직접 전달
- [ ] 파일 경로에 사용자 입력 직접 사용 (Path Traversal)

## 점검 항목 (배포 환경)

- [ ] 배포 스크립트에 `.env`, `.git`, `node_modules` 등 제외 처리 여부
- [ ] 배포 접속 정보(호스트, 비밀번호, 키)가 코드/스크립트에 하드코딩 여부
- [ ] 배포 제외 파일이 서버에 노출되지 않는지 확인 (`.git/`, `.env`, `node_modules/`)
- [ ] 배포 전/후 파일 목록 기록 여부 (변경 추적)
