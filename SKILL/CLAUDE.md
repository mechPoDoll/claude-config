# SKILL 디렉터리 규칙

## 목적
이 디렉터리는 Figma 디자인 → 분석 문서 → 개발 → 검증까지의 전체 워크플로우 산출물을 관리합니다.
초기에는 이 CLAUDE.md만 존재하며, 워크플로우 진행에 따라 아래 규칙대로 파일을 생성해 나갑니다.

> 상태 아이콘은 `~/.claude/CLAUDE.md`(글로벌)의 **공통 상태 아이콘 규칙**을 따릅니다.
> Figma 작업 시 `~/.claude/CLAUDE.md`(글로벌)의 **Figma 작업 워크플로우**(행동 원칙)를 준수합니다.
> 이 문서는 그 원칙에 따라 생성되는 **산출물의 형식과 규칙**을 정의합니다.

---

## 워크플로우 개요

```
[1] Figma 스크린샷 수집
 → [2] 스크린샷별 분석 MD 생성 + index.md 정리
  → [3] 개발 중 디자인 검증 (스크린샷·MD 재확인)
   → [4] 디자인 가이드/퍼블리싱 참고 반영
    → [5] 퍼블 ↔ 개발소스 일치도 점수화
```

## ⚠️ 파일 생성 전 필수 체크리스트

**skill/ 디렉터리에 파일을 생성·수정하기 전에 반드시 아래를 확인할 것:**

> 이 체크리스트는 "읽었다"가 아니라 "적용했다"를 검증하기 위한 것이다.
> 파일을 Write/Edit 하기 직전에 아래 항목을 하나씩 확인하고, 미충족 시 작성을 중단한다.

### 1단계: 문서 유형 판별
- [ ] 생성하려는 문서가 아래 유형 중 어디에 해당하는지 판별했는가?
  - `NN_섹션명.md` (신규 기능/메뉴 명세)
  - `CHANGE_*.md` (기존 기능 변경 요청)
  - `00_OVERVIEW.md` / `DB_SCHEMA.md` / `index.md`
- [ ] 해당 유형의 **템플릿 형식**을 아래에서 확인했는가?

### 2단계: 필수 섹션 확인
- [ ] 해당 유형의 **MUST 섹션**이 모두 초안에 포함되어 있는가?
- [ ] 빠진 MUST 섹션이 있다면, 정보 부족 시 `{{TODO: 확인 필요}}` 플레이스홀더를 넣었는가?

### 3단계: 형식 우선순위
- [ ] **이 CLAUDE.md의 템플릿 형식 > 기존 파일의 형식** 우선순위를 지켰는가?
- [ ] 기존에 비슷한 파일이 있더라도, 그 파일의 형식을 따라하지 않고 이 CLAUDE.md 템플릿을 따랐는가?

### 4단계: index.md 갱신
- [ ] index.md가 존재하면 새 항목을 추가/갱신했는가?

## 파일 명명 규칙

워크플로우 진행 중 파일을 생성할 때 아래 규칙을 따를 것:

### 스킬 문서
| 파일명 패턴 | 용도 | 예시 |
|------------|------|------|
| `00_OVERVIEW.md` | 프로젝트 전체 IA, 파일 매핑, 공통 패턴 | 항상 최초 생성 |
| `NN_섹션명.md` | 메뉴/기능별 명세서 (NN = 메뉴번호) | `01_DASHBOARD.md`, `08_CUSTOMER_CENTER.md` |
| `CHANGE_MMDD.md` | 기존 기능 변경 요청 명세 | `CHANGE_0317.md`, `CHANGE_0415.md` |
| `DB_SCHEMA.md` | DB 스키마 정의, ER 다이어그램, 명명 규칙 | 테이블/컬럼 전체 정리 |
| `index.md` | 마일스톤 트래커 | 화면 ↔ 파일 ↔ 상태 매핑 |
| `migration_*.sql` | DB 스키마 변경 SQL (필요 시) | `migration_faq_cate.sql` |

### 사용자(프론트) 문서 — 관리자/사용자 구분이 있는 프로젝트에서만 사용
| 파일명 패턴 | 용도 | 예시 |
|------------|------|------|
| `user_index.md` | 사용자 화면 트래커 | 퍼블 ↔ 개발 매핑 + 상태 |
| `USER_NN_섹션명.md` | 화면별 퍼블·백엔드 연동 명세 | `USER_01_MAIN.md`, `USER_07_MYPAGE.md` |

> 관리자/사용자 구분이 없는 프로젝트는 `NN_섹션명.md`만 사용합니다.
> 위 파일들은 워크플로우 진행 중 필요할 때 직접 작성하여 생성합니다.

## Step 1. Figma 스크린샷 수집

Figma REST API (또는 MCP get_screenshot)로 기획서·디자인의 모든 프레임을 스크린샷으로 추출합니다.

- 기획서와 디자인이 별도 파일이면 각각 수집하여 구분 저장
- Figma 참조 정보(fileKey, pageId, frameId)를 스킬 문서에 기록

### 전체 기획서 기반 스킬 작성 시 프레임 수집 절차

> **get_metadata로 하위 노드를 탐색할 수 없는 경우가 많으므로, `use_figma`를 우선 사용한다.**

1. **`use_figma`로 페이지·프레임 목록 추출** — Plugin API 코드를 실행하여 파일의 모든 페이지 목록 + 각 페이지의 최상위 프레임 목록(이름, node-id)을 가볍게 추출
2. **프레임별 상세 조회** — 추출된 목록을 기반으로 `get_design_context` 또는 `get_screenshot`을 프레임별로 순차 호출
3. **전체 종합 정리** — 수집된 정보를 종합하여 스킬 문서 작성

```javascript
// Step 1 예시: use_figma 플러그인 코드
const pages = figma.root.children;
const result = pages.map(page => ({
  name: page.name,
  id: page.id,
  frames: page.children
    .filter(node => node.type === 'FRAME' || node.type === 'SECTION')
    .map(frame => ({ name: frame.name, id: frame.id, type: frame.type }))
}));
figma.ui.postMessage(JSON.stringify(result, null, 2));
```

## Step 2. 스크린샷별 분석 MD 생성

각 화면(프레임)을 분석하여 스킬 문서를 작성합니다. 이 단계가 핵심 산출물입니다.

### 00_OVERVIEW.md 작성 형식

프로젝트 시작 시 가장 먼저 생성:

```markdown
# 프로젝트 개요

## 프로젝트 정보
| 항목 | 내용 |
|------|------|
| 프로젝트명 | {{프로젝트명}} |
| 클라이언트 | {{클라이언트}} |
| 플랫폼 | {{웹/앱/관리자 등}} |
| 기술 스택 | {{언어, DB, 프레임워크}} |
| Figma | fileKey: {{키}}, pageId: {{ID}} |

## IA (Information Architecture)
{{메뉴 트리 구조}}

## 파일 매핑
| 메뉴 | 화면 | 파일 경로 |
|------|------|----------|

## 공통 패턴
{{프로젝트에서 반복되는 코드 패턴 — include, 라우팅, 권한 체크 등}}

## AI 개발 지침
{{보안 규칙, 코딩 컨벤션 등 주의사항}}
```

### 스킬 문서 (NN_섹션명.md) 작성 형식

> 각 섹션에 **MUST** / **OPTIONAL** 을 표기. MUST 섹션은 정보 부족 시 `{{TODO: 확인 필요}}`로 표기하되 섹션 자체를 생략하지 말 것.

```markdown
# NN. 섹션명 (Page X)

## 페이지 개요                          ← MUST
| 항목 | 내용 |
|------|------|
| 페이지명 | {{페이지명}} |
| Figma 프레임 | {{frameId}} |
| 관련 파일 | {{파일 경로}} |
| 접근 권한 | {{권한 조건}} |

## 기능 요약                            ← MUST
{{1~2문장 설명}}

## ASCII 와이어프레임                    ← MUST
{{텍스트 기반 UI 표현}}

## 데이터베이스 테이블 구조               ← MUST (DB 연동 시)
{{관련 테이블 정의}}

## 화면별 상세

### N-1. 하위화면명

#### ASCII 와이어프레임                  ← MUST
{{해당 화면 UI}}

#### 필터 조건                           ← MUST (필터 있는 화면)
| 필터명 | 필터 ID | 옵션 | DB 매핑 |
|--------|---------|------|---------|

#### 목록 컬럼 정의                      ← MUST (목록 화면)
| # | 컬럼명 | DB 필드 | 비고 |
|---|--------|---------|------|

#### 데이터 쿼리                         ← OPTIONAL
{{해당 화면의 조회/처리 쿼리 또는 API 호출 패턴}}

#### 버튼 기능                           ← MUST (버튼 있는 화면)
| 버튼 | 동작 |
|------|------|
```

#### NN_섹션명.md MUST / OPTIONAL 요약

| 섹션 | 조건 |
|------|------|
| 페이지 개요 | **항상 MUST** |
| 기능 요약 | **항상 MUST** |
| ASCII 와이어프레임 | **항상 MUST** |
| 데이터베이스 테이블 구조 | DB 연동 시 MUST |
| 필터 조건 | 필터 있는 화면이면 MUST |
| 목록 컬럼 정의 | 목록 화면이면 MUST |
| 데이터 쿼리 | OPTIONAL (복잡한 쿼리일 때 권장) |
| 버튼 기능 | 버튼 있는 화면이면 MUST |

## Step 3. 개발 중 디자인 검증

AI가 개발 중 디자인과 다르게 구현하고 있다고 판단될 때:

1. 해당 화면의 **스킬 문서**(NN_섹션명.md)의 와이어프레임·필터·컬럼 정의를 재확인
2. Figma **스크린샷을 다시 캡처**하여 현재 구현과 비교
3. 불일치 항목을 구체적으로 명시하고 수정

검증 지시 예시:
> *"FAQ 목록 구현이 디자인과 다릅니다. `08_CUSTOMER_CENTER.md`의 8-3 와이어프레임과 필터 조건을 다시 확인하세요."*

## Step 5. 퍼블 ↔ 개발소스 일치도 점수화

개발 완료 후 퍼블리싱 소스와 개발 소스를 비교하여 점수 산출:

### 평가 항목
| 항목 | 배점 | 설명 |
|------|------|------|
| 레이아웃 구조 | 25 | HTML 구조, 배치 일치도 |
