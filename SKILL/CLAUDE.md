# SKILL 디렉터리 규칙

## 목적
이 디렉터리는 Figma 디자인 → 분석 문서 → 개발 → 검증까지의 전체 워크플로우 산출물을 관리합니다.
초기에는 이 CLAUDE.md만 존재하며, 워크플로우 진행에 따라 아래 규칙대로 파일을 생성해 나갑니다.

> 상태 아이콘은 `~/.claude/CLAUDE.md`(글로벌)의 **공통 상태 아이콘 규칙**을 따릅니다.
> Figma 작업 시 `~/.claude/CLAUDE.md`(글로벌)의 **Figma 작업 워크플로우**(행동 원칙)를 준수합니다.
> 이 문서는 그 원칙에 따라 생성되는 **산출물의 형식과 규칙**을 정의합니다.

---

## 워크플로우 개요

```
[1] Figma 스크린샷 수집
 → [2] 스크린샷별 분석 MD 생성 + index.md 정리
  → [3] 개발 중 디자인 검증 (스크린샷·MD 재확인)
   → [4] 디자인 가이드/퍼블리싱 참고 반영
    → [5] 퍼블 ↔ 개발소스 일치도 점수화
```

---

## 파일 명명 규칙

워크플로우 진행 중 파일을 생성할 때 아래 규칙을 따를 것:

### 스킬 문서
| 파일명 패턴 | 용도 | 예시 |
|------------|------|------|
| `00_OVERVIEW.md` | 프로젝트 전체 IA, 파일 매핑, 공통 패턴 | 항상 최초 생성 |
| `NN_섹션명.md` | 메뉴/기능별 명세서 (NN = 메뉴번호) | `01_DASHBOARD.md`, `08_CUSTOMER_CENTER.md` |
| `DB_SCHEMA.md` | DB 스키마 정의, ER 다이어그램, 명명 규칙 | 테이블/컬럼 전체 정리 |
| `index.md` | 마일스톤 트래커 | 화면 ↔ 파일 ↔ 상태 매핑 |
| `migration_*.sql` | DB 스키마 변경 SQL (필요 시) | `migration_faq_cate.sql` |

### 사용자(프론트) 문서 — 관리자/사용자 구분이 있는 프로젝트에서만 사용
| 파일명 패턴 | 용도 | 예시 |
|------------|------|------|
| `user_index.md` | 사용자 화면 트래커 | 퍼블 ↔ 개발 매핑 + 상태 |
| `USER_NN_섹션명.md` | 화면별 퍼블·백엔드 연동 명세 | `USER_01_MAIN.md`, `USER_07_MYPAGE.md` |

> 관리자/사용자 구분이 없는 프로젝트는 `NN_섹션명.md`만 사용합니다.
> 위 파일들은 워크플로우 진행 중 필요할 때 직접 작성하여 생성합니다.

---

## Step 1. Figma 스크린샷 수집

Figma REST API (또는 MCP get_screenshot)로 기획서·디자인의 모든 프레임을 스크린샷으로 추출합니다.

- 기획서와 디자인이 별도 파일이면 각각 수집하여 구분 저장
- Figma 참조 정보(fileKey, pageId, frameId)를 스킬 문서에 기록

---

## Step 2. 스크린샷별 분석 MD 생성

각 화면(프레임)을 분석하여 스킬 문서를 작성합니다. 이 단계가 핵심 산출물입니다.

### 00_OVERVIEW.md 작성 형식

프로젝트 시작 시 가장 먼저 생성:

```markdown
# 프로젝트 개요

## 프로젝트 정보
| 항목 | 내용 |
|------|------|
| 프로젝트명 | {{프로젝트명}} |
| 클라이언트 | {{클라이언트}} |
| 플랫폼 | {{웹/앱/관리자 등}} |
| 기술 스택 | {{언어, DB, 프레임워크}} |
| Figma | fileKey: {{키}}, pageId: {{ID}} |

## IA (Information Architecture)
{{메뉴 트리 구조}}

## 파일 매핑
| 메뉴 | 화면 | 파일 경로 |
|------|------|----------|

## 공통 패턴
{{프로젝트에서 반복되는 코드 패턴 — include, 라우팅, 권한 체크 등}}

## AI 개발 지침
{{보안 규칙, 코딩 컨벤션 등 주의사항}}
```

## Step 3. 개발 중 디자인 검증

AI가 개발 중 디자인과 다르게 구현하고 있다고 판단될 때:

1. 해당 화면의 **스킬 문서**(NN_섹션명.md)의 와이어프레임·필터·컬럼 정의를 재확인
2. Figma **스크린샷을 다시 캡처**하여 현재 구현과 비교
3. 불일치 항목을 구체적으로 명시하고 수정

검증 지시 예시:
> *"FAQ 목록 구현이 디자인과 다릅니다. `08_CUSTOMER_CENTER.md`의 8-3 와이어프레임과 필터 조건을 다시 확인하세요."*

---

## Step 4. 디자인 가이드 · 퍼블리싱 참고

- 퍼블리싱 소스가 있으면 USER_NN 문서에 **퍼블 파일 ↔ 개발 파일 매핑** 기록
- 퍼블리싱의 헤더/푸터/메뉴 구조, 사용 라이브러리를 문서에 명시
- 개발 시 퍼블리싱 마크업/스타일을 우선 참고하여 구현

---

## Step 5. 퍼블 ↔ 개발소스 일치도 점수화

개발 완료 후 퍼블리싱 소스와 개발 소스를 비교하여 점수 산출:

### 평가 항목
| 항목 | 배점 | 설명 |
|------|------|------|
| 레이아웃 구조 | 25 | HTML 구조, 배치 일치도 |
| 스타일 정확도 | 25 | 색상, 폰트, 간격, 크기 등 디자인 토큰 준수 |
| 반응형 대응 | 20 | 브레이크포인트별 레이아웃 일치도 |
| 인터랙션 구현 | 15 | 호버, 클릭, 트랜지션, 애니메이션 |
| 접근성 · 시맨틱 | 15 | 시맨틱 태그, ARIA, 키보드 네비게이션 |

### 점수 기준
| 점수 | 판정 |
|------|------|
| 90~100 | 퍼블과 거의 동일, 수정 불필요 |
| 70~89 | 대체로 일치, 미세 조정 필요 |
| 50~69 | 구조는 맞으나 디테일 차이 다수 |
| 50 미만 | 재작업 필요 |

### 결과 보고 형식
```
## 퍼블 ↔ 개발 일치도 점수표
| 화면 | 레이아웃 | 스타일 | 반응형 | 인터랙션 | 접근성 | 합계 |
|------|----------|--------|--------|----------|--------|------|
| 메인 | 25 | 23 | 18 | 13 | 12 | 91 |
```
