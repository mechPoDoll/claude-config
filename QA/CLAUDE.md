# QA 디렉터리 규칙

> ⚠️ **이 파일은 사람용 참고 문서입니다.**
> 에이전트 자동 로딩이 보장되지 않으므로, 에이전트용 핵심 규칙은 루트 `CLAUDE.md`에 직접 포함되어 있습니다.
> 이 파일은 QA 작업의 상세 형식과 절차를 사람이 참고하기 위한 문서입니다.

---

## 목적
이 디렉터리는 프로젝트의 품질 보증(QA) 산출물을 관리합니다.
초기에는 이 CLAUDE.md만 존재하며, QA 진행에 따라 아래 규칙대로 파일을 생성해 나갑니다.

> 상태 아이콘은 `~/.claude/CLAUDE.md`(글로벌)의 **공통 상태 아이콘 규칙**을 따릅니다.

---

## QA 진행 워크플로우

```
[1] 보안 감사 → [2] 조치 → [3] 테스트 작성·실행 → [4] 배포 → [5] 검증
```

각 단계에서 생성되는 산출물은 이 디렉터리에 아래 명명 규칙에 따라 저장합니다.

---

## 파일 명명 규칙

QA 작업 중 파일을 생성할 때 아래 규칙을 따를 것:

| 유형 | 파일명 패턴 | 예시 |
|------|------------|------|
| QA 이력 마스터 | `index.md` | 전체 QA 현황 요약, 보고서 링크 모음 |
| 보안감사 보고서 | `qa_YYYYMMDD_v#.md` | `qa_20260318_v1.md` |
| 주제별 보고서 | `qa_YYYYMMDD_v#_주제.md` | `qa_20260318_v2_db_tuning.md` |
| 레거시 체크리스트 | `legacy_checklist.md` | 레거시 프로젝트 초기 점검용 |
| E2E 테스트 스크립트 | `e2e-범위.*` | `e2e-test.mjs`, `e2e-security.py` 등 |
| 비주얼 테스트 스크립트 | `visual-test.*` | 스크린샷 비교 |
| 테스트 리포트 | `범위-report.html` | `e2e-report.html` |
| 배포 스크립트 | `deploy.*` / `deploy_모듈명.*` | `deploy.sh`, `deploy.py` |
| 스크린샷 | `screenshots/` | 비주얼 테스트 베이스라인 |
| 배포 백업 | `backups/YYYYMMDD_HHMMSS/` | 배포 전 자동 백업 |

> 위 파일들은 QA 진행 과정에서 필요할 때 직접 작성하여 생성합니다.
> 처음부터 존재하는 것이 아닙니다.

## index.md 작성 규칙

QA를 시작하면 가장 먼저 `index.md`를 생성하고, 이후 보고서가 추가될 때마다 갱신할 것:

```markdown
# QA 이력

## 프로젝트 정보
| 항목 | 내용 |
|------|------|
| 프로젝트 | {{프로젝트명}} |
| 대상 URL | {{URL}} |
| 환경 | {{기술 스택}} |

## QA 보고서 목록
| # | 날짜 | 파일 | 내용 | 상태 |
|---|------|------|------|------|
| 1 | YYYY-MM-DD | [파일명](파일명) | 설명 | ✅/🔄/⬜ |

## 취약점 현황 요약
| 등급 | 항목수 | 조치완료 | 미조치 |
|------|--------|----------|--------|
| CRITICAL | - | - | - |
| HIGH | - | - | - |
| MEDIUM | - | - | - |
```

## 보고서 작성 형식

### 보안감사 보고서
```markdown
# 보안 감사 보고서

| 항목 | 내용 |
|------|------|
| 일자 | YYYY-MM-DD |
| 버전 | v# |
| 대상 | {{대상 시스템}} |
| 환경 | {{기술 스택}} |

## CRITICAL
| # | 항목 | 영향 범위 | 상태 |
|---|------|----------|------|
| 1 | 항목명 | N개 파일 | ❌/✅ |

### C-1. 항목 상세
- **위험**: 구체적 위험 설명
- **영향 범위**: 파일 목록 또는 개수
- **조치 방법**: 수정 코드/절차
- **조치 결과**: 수정 후 상태

## HIGH / MEDIUM / LOW
(동일 형식)

## 조치 우선순위 로드맵
| 단계 | 내용 | 예상 기간 |
|------|------|----------|
| Stage 1 | 긴급 조치 (백도어, 파일업로드, 경로탐색) | 즉시 |
| Stage 2 | 주입 취약점 (SQLi, XSS, 평문 비밀번호) | 1~2일 |
| Stage 3 | 인프라 (세션, CSRF, 인증, 시크릿 분리) | 1~2일 |
| Stage 4 | 성능 (DB 인덱스, 슬로우쿼리) | 1일 |
| Stage 5 | 추가 경화 | 지속 |
```

### DB 튜닝 보고서
```markdown
# DB 튜닝 보고서

| 항목 | 내용 |
|------|------|
| DB | {{종류 및 버전}} |
| 테이블 수 | N개 |

## 1. 인덱스 (CRITICAL) — 누락 인덱스 + 추가 구문
## 2. N+1 쿼리 (HIGH) — 문제 함수 + JOIN 리팩토링
## 3. 집계 쿼리 (HIGH) — 비정규화/캐싱 권장
## 4. 페이지네이션 (MEDIUM) — LIMIT 없는 조회 목록
## 5. DB 설정 (MEDIUM) — 슬로우쿼리 로그, 버퍼 풀 등
## 6. 데이터 타입 (LOW) — 타입/문자셋 불일치
```

## E2E 테스트 작성 지침

필요 시 프로젝트 환경에 맞는 E2E 테스트를 직접 작성합니다.
테스트 도구는 프로젝트에 따라 선택합니다 (Playwright, Cypress, Selenium, Puppeteer 등).

### 작성 원칙
- 테스트 데이터에는 `[E2E_TEST]` 접두사를 붙여 실제 데이터와 구분
- 테스트별 결과를 수집하여 리포트로 출력
- 가능하면 headless / headed 모드 모두 지원

### 테스트 분류 기준
| 분류 | 목적 | 파일명 예시 |
|------|------|------------|
| 기능 CRUD | 핵심 화면별 생성/수정/삭제 검증 | `e2e-test.*` |
| 전체 커버리지 | 전 메뉴/페이지 테스트 | `e2e-full.*` |
| 보안 | 비로그인 차단, CSRF 검증, XSS 필터 | `e2e-security.*` |
| 비주얼 리그레션 | 스크린샷 비교로 UI 변경 감지 | `visual-test.*` |

### 리포트 구조
- 상단: PASS / FAIL / SKIP 카운트 대시보드
- 본문: 테스트 스위트별 그룹 → 각 테스트 상태/이름/상세
- 실패 항목: 하이라이트 구분

## 배포 스크립트 작성 지침

필요 시 배포 자동화 스크립트를 직접 작성합니다.
배포 방식은 프로젝트에 따라 선택합니다 (FTP, SSH, CI/CD, 컨테이너 등).

### 지원 모드
| 모드 | 설명 |
|------|------|
| `full` | 소스 + DB 전체 배포 |
| `src` | 소스만 배포 |
| `db` | DB만 배포 |
| `diff` | VCS diff 기반 변경분만 배포 |
| `rollback` | 백업에서 복원 |

### 필수 요구사항
- 배포 전 원격 백업 자동 생성 (`backups/YYYYMMDD_HHMMSS/`)
- 배포 후 핵심 페이지/엔드포인트 상태 자동 검증
- 확인 프롬프트 (y/N) 후 실행

## 레거시 프로젝트 체크리스트

레거시 프로젝트 인수 시 `legacy_checklist.md`를 작성하여 5단계 점검:

| 단계 | 내용 |
|------|------|
| 1. 환경 구축 | 런타임/DB 설정, 의존성 설치, .env 분리 |
| 2. 소스 에러 | 문법 오류, 런타임 버전 호환, deprecated API 수정 |
| 3. 보안 감사 | 백도어, 파일업로드, SQLi, XSS, CSRF, 세션, 인증 |
| 4. DB 튜닝 | 인덱스, N+1, 페이지네이션, 슬로우쿼리 |
| 5. 운영 준비 | 모니터링, 배포, 백업 자동화 |
